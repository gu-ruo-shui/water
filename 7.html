编译链接加载

变量有两重信息: 符号和类型
类型限制编译器该用哪些汇编指令
符号限制链接器引用哪个地址

链接:将定义符号的地址赋值给引用符号地方, 动态链接将符号处理推迟到加载过程中

加载:将代码和数据复制进内在,然后跳转到函数入口执行

九曲阑干-静态库的链接过程

手动进行静态链接

gcc -c main.c 

gcc -static main.c -Wl,--trace

link_files=(/lib/x86_64-linux-gnu/crt1.o /lib/x86_64-linux-gnu/crti.o /usr/lib/gcc/x86_64-linux-gnu/11/crtbeginT.o /usr/lib/gcc/x86_64-linux-gnu/11/libgcc.a /usr/lib/gcc/x86_64-linux-gnu/11/libgcc_eh.a /usr/lib/x86_64-linux-gnu/libc.a /usr/lib/gcc/x86_64-linux-gnu/11/crtend.o /usr/lib/x86_64-linux-gnu/crtn.o)

ld main.o $link_files

for file in $link_files; do echo $file; readelf -s $file | grep "_Unwind_Resume"; echo $file; done

ld main.o $link_files /usr/lib/gcc/x86_64-linux-gnu/11/libgcc.a /usr/lib/gcc/x86_64-linux-gnu/11/libgcc_eh.a /usr/lib/x86_64-linux-gnu/libc.a

# 静态链接需要加 start-group

ld main.o /lib/x86_64-linux-gnu/crt1.o /lib/x86_64-linux-gnu/crti.o /usr/lib/gcc/x86_64-linux-gnu/11/crtbeginT.o --start-group /usr/lib/gcc/x86_64-linux-gnu/11/libgcc.a /usr/lib/gcc/x86_64-linux-gnu/11/libgcc_eh.a /usr/lib/x86_64-linux-gnu/libc.a --end-group /usr/lib/gcc/x86_64-linux-gnu/11/crtend.o /usr/lib/x86_64-linux-gnu/crtn.o

手动进行动态链接

gcc main.o -v

ld main.o /usr/lib/x86_64-linux-gnu/libc.so --dynamic-linker /lib64/ld-linux-x86-64.so.2 /lib/x86_64-linux-gnu/crt1.o /lib/x86_64-linux-gnu/crti.o /lib/x86_64-linux-gnu/crtn.o

If the linker cannot recognize the format of an object file, it will assume that it is a linker script.

-- man ld

链接将符号的地址赋值给引用符号地方

指针与数组(symbol)

  • extern char* foo

也会有坏处

由此带来的重名符号失效,

  • 静态链接符号冲突
  • 动态链接符号覆盖

If a symbol is defined in two different libraries gcc will use the first one it finds and ignore the second one unless the second one is included in an object file which gets included for some other reason. - link

基于以上几点

  • 尽可能的使用 static
  • library design:prefix + symbol

解决方法

  • 源码 rename
  • 二进制 hack rename
